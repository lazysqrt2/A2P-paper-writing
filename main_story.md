# A2P 学术主线 (Main Story)

> 定稿 2026-06-15。本文件是 A2P 的**核心学术定位**:正面主线见谁都能讲、不点名任何对手;竞品只活在文末的"动机与防守"层。写 Intro / Method / Experiments 或做 rebuttal 时以此为准。

---

## 一句话版本

> A2P 研究的不是"让 LLM 生成练手项目",而是:**如何把任意知识材料*编译*成一个可执行、可学习、可校准的项目环境,让学习目标通过学生亲手完成的工程动作被掌握。**

最压缩的英文版:

> **A2P studies how to compile arbitrary knowledge into executable project work, and how to leave that work unfinished in exactly the way that causes learning.**

---

## 正面主线(不点名任何对手)

### 1. 我们要回答的问题

给定任意输入——一篇论文、一个代码库、一份教程、一个数据集——里面藏着一套知识、技能、概念之间的依赖关系,以及易错点。我们要回答的是:

> 怎么把它自动变成一个**能跑起来、并且学生亲手做一遍就能真正掌握那套知识**的项目环境?

关键词不是 **generation**,而是 **compilation(编译)**。

### 2. 一个让任何人秒懂的洞察:学习发生在"动作"里,不在"内容"里

学生真正学会 RAG,不是因为读过一段 RAG 的解释,而是因为他在一个项目里亲手处理过 chunking、embedding、retrieval failure、hallucination mitigation。学生真正学会后端认证,也不是因为看懂了 JWT 文档,而是因为他**实现、测试、调试过**整条认证流程。

所以一个项目能不能教会人,**不写在它的代码里,也不写在它的说明文字里**——而写在一个**对照关系**上:

> 输入材料里的概念结构 ←→ 项目逼着学生亲手做的那些工程动作。

只有当项目要学生做的事,**恰好覆盖、并按正确依赖顺序激活**了输入里那套概念,做完它才等于学会。我们把这个"让概念变成学生动作"的过程叫 **embodiment**。

### 3. compilation:这条对照关系是一条可追踪的编译链

A2P 要把输入材料里的概念,逐级编译成学生的动作和验证手段:

```
source concept
  → learning objective
  → project requirement
  → student action
  → scaffold / test / checkpoint
```

这把"对照关系"从一张静态的表,变成一条**有向的、可追踪的编译链**。这也是为什么 A2P 是一个 method/system,而不是一句 prompt。

### 4. 为什么这是个真问题,而且结构上很硬

既然"学会"是输入与项目之间的一个**对照关系**,那么要做出一个真能教会人的项目,就必须**同时拿出这条关系的两端**:

- 一端是**"该学什么"**——从任意输入里抽出的概念/技能依赖图:有哪些概念、谁依赖谁、哪些核心哪些枝节、对这个学习者哪些是恰好的挑战;
- 另一端是**"项目逼你做什么"**——它有没有逐项覆盖、顺序对不对、难度合不合适、有没有把答案漏出来。

这里有一条**难以反驳的逻辑**,是整个故事最硬的地方:

> 一个**关系**要能**被核对**,必须把**两端都写下来**。一次性生成一个项目,只写下了**一端(那个仓库)**;另一端(输入里到底该学什么、什么顺序)从未被显式产出。**而你无法核对一个只写下了一半的对照关系。**

所以"这个项目到底教没教会人"不是 prompt 写得好不好的问题,而是**结构问题**:不把另一端显式写出来并对齐,在逻辑上就**无法验证**,只能"断言"。

因此 A2P 把这些一直隐式的东西**显式化**,不是"多做了点好功能",而是**不这么做就不可能验证**:

- 输入材料里的核心知识,是否真的被转成了学生要完成的动作;
- 哪些部分由系统给出,哪些部分该学生自己做;
- 缺失的代码是**教学设计**,还是随便留空;
- hint 有没有泄露答案;
- 测试是不是在验证**学习目标**,而不只是软件行为;
- 换一个学习者水平,项目该如何相应变化;
- 老师能否检查和修改这个设计。

### 5. 最锋利的学术点:controlled incompletion

PBL 项目既不是**完整 repo**,也不是**残缺 repo**,而是 **controlled incompletion(受控的未完成)**:

- 它**必须能跑**,但又**不能替学生完成**该由学生学习的部分;
- 它**必须留白**,但留白不能太高导致**过载**,也不能太低变成 **busywork**;
- 它**必须来自输入材料**,而不能只是把材料**包装成说明文字**;
- 它**必须适配学习者**,因为同一个项目对新手、中级、高手应该留下**不同的认知动作**。

> 核心学术主张:**未完成不是失败,而是一个可设计、可追踪、可评测的教学对象。**

### 6. 核心产物:Pedagogical Repository Contract

A2P 的核心产物**不是"一个仓库"**,而是一份显式的、随学习者校准的、可被检查和编辑的 **Pedagogical Repository Contract**。它规定:

```
这个知识点要学什么;
它对应项目里的哪个任务;
这个任务由系统完成、学生完成,还是 scaffolded;
为什么这个 learner profile 适合这样分配;
用什么 test / rubric / checkpoint 来验证它。
```

有了这份 contract,生成才有了一个可核对的"另一端";没有它,任何生成出来的项目都只能被"信我",不能被"验证"。

### 7. 三个不可替代点

1. **Anything**:任意输入不是被直接变成文本内容,而是被**编译成项目工作**。
2. **PBL**:学习不是阅读生成物,而是在**可执行环境中完成真实任务**。
3. **A2P 的研究贡献**:未完成不是失败,而是**可设计、可追踪、可评测**的教学对象。

### 8. 评价自然成立

- **Source faithfulness**:项目是否覆盖输入材料的核心概念。
- **Embodiment**:概念是否真的变成了学生要做的动作。
- **Controlled incompletion**:缺口是否只出现在学习目标处。
- **Leakage**:starter code / hint / test 是否泄露答案。
- **Profile sensitivity**:换 learner profile 后,任务所有权(谁来做)是否合理变化。
- **Traceability**:每个任务、hint、测试是否都能追溯到 source concept 和 learning objective。

### 贴墙版

> 一个项目能不能教会你,不在代码里,而在"该学的东西"和"项目逼你做的事"之间那条对照关系上。A2P 把任意知识材料**编译**成这条关系,显式写出两端、对齐、并验证,交出一个**既真能跑、又被验证过确实能教会人**的项目环境——而它的精髓,是把"未完成"做成一个可设计、可评测的教学对象。

## 动机与防守层(竞品只出现在这里,不进主线)

### 现状:LLM 已经很会做两类事

一类是**教育内容生成**:把知识变成 slides、lesson plan、quiz、讲解、对话。局限在于,这些主要让学生"看、听、答"——产物是文本,不用真的跑起来。

另一类是**编程 agent**:把需求变成能跑的代码。局限在于,它的目标是把所有缺口**填平**、尽快交付一个完整 artifact。

PBL 需要的是**第三种能力**:把知识变成**可做的项目**——一个能跑、却又在恰当之处为学生留白的可执行环境。A2P 占的就是这个位置。

### 防守:"这不就是给某个编程 agent 喂 prompt 让它生成练手项目吗?"

直接 prompt 通常只会生成一个**看起来合理**的项目建议。它可能不错,但它**没有保证**第 4 节列出的任何一条(核心知识有没有变成动作、缺口是不是设计、hint 有没有泄漏、测试验不验证学习目标、换 learner 怎么变、老师能不能改)。

把这个质疑逼成一个**两难**:

- **要么**对方真把"先列出该学什么 → 逐项绑定到项目动作 → 验证覆盖/顺序/难度/不泄漏"都做出来——那他已经**用 prompt 把 A2P 的方法重做了一遍**,等于承认值钱的是这套**方法与 contract**,编程 agent 只是底层**实现引擎**(Codex、Claude Code、OpenHands 都可以当 generator,A2P 研究的从不是哪个 agent 更会写代码);
- **要么**他没真做——那它就会犯那些躲不掉的毛病:漏掉输入里的核心概念、顺序乱、难度失准、泄漏答案、做完也不保证学会。

而把"断言能学会"变成"验证过能学会",靠的是 A2P 的**评测机制**(覆盖度、profile sensitivity、leakage、traceability、教师在环纠正)——**这根本不是 prompt,而是研究本身**。

### 一句收尾

A2P 不需要声称"我们比某个 agent 更聪明"。它只需要说明:直接 prompt **没有定义这个问题**、没有中间 contract、没有 trace、也没有 controlled incompletion 的评价。A2P 的贡献,是把"**任意输入如何变成可项目化的学习环境**"这一过程**形式化**,并给出一个**可检查、可编辑、可评测**的生成框架。
